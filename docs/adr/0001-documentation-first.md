# ADR 0001: Documentation-first architecture process

Status: Accepted
Date: 2026-07-15

## Context

AgentMesh 涉及任务状态、持久执行、Agent 调度、A2A、MCP、权限、Artifact 和可观测等多个边界。如果先写代码，早期实现细节很容易替代领域设计，并导致协议对象、数据库实体和用户概念互相耦合。

## Decision

在运行时代码开始前：

1. 使用 L0–L3 分层描述架构。
2. 先评审 L0 系统边界和 L1 容器边界。
3. 跨模块且难逆转的决定使用 ADR。
4. 先完成最小垂直切片相关模块的 L2 设计，再开始对应实现。
5. 不要求所有模块设计完成后才开始编码，但任何代码必须有已接受的上层边界。

## Consequences

Positive:

- 减少协议、框架和业务模型之间的意外耦合。
- 便于开源贡献者理解设计意图。
- 可以逐步深入，而不是一次性进行巨型前置设计。
- 架构决策和实现变化具有可追踪历史。

Negative:

- 首个可运行 Demo 会更晚出现。
- 文档需要与实现持续同步。
- 设计评审需要明确负责人和接受标准。

## Alternatives considered

- 先实现 Demo 再补文档：反馈快，但对持久状态和协议边界风险过高。
- 一次性完成所有详细设计：会形成过度前置设计，且缺少运行反馈。
- 仅使用代码和 Issue 记录决策：难以形成连续、可阅读的系统模型。
